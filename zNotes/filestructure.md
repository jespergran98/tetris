I'm building a multiplayer Tetris game. The project is split across two repositories вЂ” a backend (handled separately) and this front-end React application. Your sole task is to design the optimal front-end file and folder structure for this project.

## Tech Stack

- **Framework** вЂ” React 19 with TypeScript 5
- **Bundler** вЂ” Vite
- **Routing** вЂ” React Router v7
- **Styling** вЂ” CSS Modules with CSS variables from `styles/global.css`
- **State** вЂ” React Context + hooks only (no Redux, no Zustand)
- **Communication** вЂ” WebSocket (real-time game events) + REST (leaderboard, room management)
- **No external game libraries** вЂ” all Tetris logic is pure TypeScript in `utils/`

---

## Backend Responsibilities (out of scope for this repo)

The backend lives in a separate repository. The front-end communicates with it over two channels:

**WebSocket (real-time):**
- Distributes the shared 7-bag piece sequence seed to all players in a room at game start
- Receives lock-event payloads from each client on every piece lock for passive cheat validation (the backend reconstructs and verifies the board state after the fact вЂ” it never sits in the critical path of gameplay)
- Redistributes opponent board snapshots to all players in the room
- Manages room lifecycle events (player joined, player left, game started, game ended)

**REST API (non-real-time):**
- Creating and joining rooms
- Fetching and submitting leaderboard scores

---

## Front-End Responsibilities

- All rendering and UI
- Running the local player's game loop with full authority вЂ” movement, gravity, rotation, collision detection, locking, line clears, and scoring are all computed entirely client-side and never wait on the backend
- On each piece lock, sending a lock-event payload (piece type, final position, resulting board state, current score) to the backend for passive validation
- Receiving the shared piece sequence seed from the backend at game start and using it to generate the local 7-bag sequence
- Displaying opponent board snapshots received from the backend as read-only previews
- Routing between screens
- Responsive layout across desktop and tablet viewports

## Game Features

**Playfield**
- 10 columns Г— 20 rows grid (the Matrix)
- Rendered as a CSS grid of cells

**Tetrominoes**
- Seven pieces: I, J, L, O, S, T, Z
- Each composed of 4 blocks with a unique colour defined as a CSS variable

**Movement & Rotation**
- Shift left/right
- Soft drop (hold key вЂ” accelerated fall)
- Hard drop (instant snap to lowest valid position)
- Rotate 90В° clockwise and counter-clockwise using the Super Rotation System (SRS) with wall kick tables

**Ghost Piece**
- Semi-transparent preview of where the active piece will land
- Rendered inside `TetrisBoard` вЂ” not a separate component

**Gravity & Speed**
- Pieces fall automatically via a `requestAnimationFrame` game loop
- Fall interval decreases as level increases

**Levels & Difficulty**
- Level = `Math.floor(linesCleared / 10) + 1`
- Speed increases with each level

**Line Clears**
- Completed rows are removed; all blocks above drop down
- Triggers score update and level recalculation

**Scoring**
- Single (1 line): 100 Г— level
- Double (2 lines): 300 Г— level
- Triple (3 lines): 500 Г— level
- Tetris (4 lines): 800 Г— level

**7-Bag Randomizer**
- All 7 tetrominoes appear once per cycle before any repeat
- In multiplayer, the backend distributes a shared seed at game start; every client generates the same sequence independently from that seed

**Hold Piece**
- Player can hold the active piece once per piece turn
- Swapping is locked until the current piece locks down

**Next Piece Preview**
- Shows the next piece in the queue

**Collision Detection**
- Prevents movement through walls, floor, and settled blocks

**Lock Down**
- Active piece locks in place when it can no longer fall
- Triggers a lock-event payload to be sent to the backend

**Game Over**
- Triggered when a new piece cannot enter the field at the spawn position

## Architecture: WebSocket Three-Layer Separation

1. **`services/websocket.service.ts`** вЂ” raw WebSocket class with no React dependency: handles connection lifecycle, message serialisation, send/receive, and reconnection logic
2. **`hooks/useWebSocket.ts`** вЂ” React hook that wraps the service: ties the connection to the component lifecycle, exposes connection status and a typed send function
3. **`hooks/useMultiplayer.ts`** вЂ” game-level hook built on top of `useWebSocket.ts`: translates incoming WebSocket messages into room and opponent game state, and exposes methods for sending lock-event payloads

---

## Structural Principles

These are strict rules вЂ” follow them exactly and justify any deviation:

- Every component lives in its own named folder: `ComponentName/ComponentName.tsx` + `ComponentName/ComponentName.module.css`
- Each folder group (`ui/`, `game/`, `layout/`, `pages/`) exports all members via a barrel `index.ts`
- Hooks contain only React logic (state, effects, refs) вЂ” all pure game calculations belong in `utils/`
- No inline type definitions in components вЂ” all shared types live in `types/`
- No hardcoded strings or magic numbers in components вЂ” all values live in `constants/`
- Route paths are defined once in `constants/routes.constants.ts` and imported everywhere they are needed
- All colours, spacing, font sizes, and timing values are CSS variables declared in `styles/global.css` вЂ” no raw values in any `.module.css` file
- Naming conventions: `PascalCase` for components and types, `camelCase` for hooks, utils, services, and constants files

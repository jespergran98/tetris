## Tech Stack

- **Framework** — React 19 with TypeScript 5
- **Bundler** — Vite
- **Routing** — React Router v7
- **Styling** — CSS Modules with CSS variables from `styles/global.css`
- **State** — React Context + hooks only (no Redux, no Zustand, no external state library)
- **Communication** — WebSocket (real-time game events) + REST (leaderboard, room management)
- **No external game libraries** — all Tetris logic is pure TypeScript in `utils/`

---

## Backend Responsibilities (out of scope for this repo)

The backend lives in a separate repository. The front-end communicates with it over two channels:

**WebSocket (real-time):**
- Distributes the shared 7-bag piece sequence seed to all players in a room at game start
- Receives lock-event payloads from each client on every piece lock for passive cheat validation (the backend reconstructs and verifies the board state after the fact — it never sits in the critical path of gameplay)
- Redistributes opponent board snapshots to all players in the room
- Manages room lifecycle events (player joined, player left, game started, game ended)

**REST API (non-real-time):**
- Creating and joining rooms
- Fetching and submitting leaderboard scores

---

## Front-End Responsibilities

- All rendering and UI
- Running the local player's game loop with full authority — movement, gravity, rotation, collision detection, locking, line clears, and scoring are all computed entirely client-side and never wait on the backend
- On each piece lock, sending a lock-event payload (piece type, final position, resulting board state, current score) to the backend for passive validation
- Receiving the shared piece sequence seed from the backend at game start and using it to generate the local 7-bag sequence
- Displaying opponent board snapshots received from the backend as read-only previews
- Routing between screens
- Responsive layout across desktop and tablet viewports
- Persisting user settings (volume, keybinds) to `localStorage`

## Application Screens & Flow

```
Start Screen  [Settings button always visible in top-right corner]
├── → Singleplayer → Game Screen (singleplayer layout)
└── → Multiplayer
    └── Lobby Screen
        ├── Host → room code generated by backend → waiting room → Game Screen (multiplayer layout)
        └── Join → enter room code → waiting room → Game Screen (multiplayer layout)

Settings Modal (accessible from Start Screen and paused Game Screen)
├── Audio tab — master volume slider
└── Controls tab — rebindable keybind list (one action per row, click to rebind)
```

**Start Screen** — two buttons: Singleplayer, Multiplayer. Settings icon button in top-right corner opens the Settings Modal.

**Lobby Screen** — Host path displays the generated room code (up to 10 players can join via this code). Join path shows a code entry input. Both paths show a waiting room listing connected players. Only the host can start the game.

**Game Screen (Singleplayer layout)** — Hold piece panel (left), main Tetris board (centre), Next piece preview + Score + Level + Lines cleared (right). Pause overlay triggered by Escape key — the Settings Modal is also accessible from within the Pause overlay. Game Over overlay on loss.

**Game Screen (Multiplayer layout)** — Player's own large board (centre), up to 9 opponent boards as small read-only snapshots surrounding it, live leaderboard sidebar. Game Over overlay shows final rankings fetched from the backend.

**Settings Modal** — Opened as an overlay (not a separate route) from both the Start Screen and the Pause overlay. Contains two tabs:
- **Audio** — a master volume slider (0–100). Value is persisted to `localStorage`.
- **Controls** — a list of every rebindable game action with its currently assigned key. Clicking a row enters a listening state; the next key press is captured and assigned. Keybinds are persisted to `localStorage`. Defaults are defined in `constants/keybinds.constants.ts`.

## Game Features

**Playfield**
- 10 columns × 20 rows grid (the Matrix)
- Rendered as a CSS grid of cells

**Tetrominoes**
- Seven pieces: I, J, L, O, S, T, Z
- Each composed of 4 blocks with a unique colour defined as a CSS variable

**Movement & Rotation**
- Shift left/right
- Soft drop (hold key — accelerated fall)
- Hard drop (instant snap to lowest valid position)
- Rotate 90° clockwise and counter-clockwise using the Super Rotation System (SRS) with wall kick tables

**Ghost Piece**
- Semi-transparent preview of where the active piece will land
- Rendered inside `TetrisBoard` — not a separate component

**Gravity & Speed**
- Pieces fall automatically via a `requestAnimationFrame` game loop
- Fall interval decreases as level increases

**Levels & Difficulty**
- Level = `Math.floor(linesCleared / 10) + 1`
- Speed increases with each level

**Line Clears**
- Completed rows are removed; all blocks above drop down
- Triggers score update and level recalculation

**Scoring**
- Single (1 line): 100 × level
- Double (2 lines): 300 × level
- Triple (3 lines): 500 × level
- Tetris (4 lines): 800 × level

**7-Bag Randomizer**
- All 7 tetrominoes appear once per cycle before any repeat
- In multiplayer, the backend distributes a shared seed at game start; every client generates the same sequence independently from that seed

**Hold Piece**
- Player can hold the active piece once per piece turn
- Swapping is locked until the current piece locks down

**Next Piece Preview**
- Shows the next piece in the queue

**Collision Detection**
- Prevents movement through walls, floor, and settled blocks

**Lock Down**
- Active piece locks in place when it can no longer fall
- Triggers a lock-event payload to be sent to the backend

**Game Over**
- Triggered when a new piece cannot enter the field at the spawn position

## Settings System

- All rebindable actions and their default keys are defined in `constants/keybinds.constants.ts`. Components and hooks must never hardcode key strings.
- Volume and keybind state is managed in `context/SettingsContext.tsx` and persisted to `localStorage` on every change.
- `hooks/useSettings.ts` is the single consumer-facing hook — it reads from `SettingsContext` and exposes typed getters and setters.
- `hooks/useInput.ts` reads the current keybind map from `useSettings` at runtime so that rebinding takes effect immediately without a page reload.
- Audio playback (if added later) must read volume from `SettingsContext`. No component or hook may read `localStorage` directly for settings — all access goes through the context.

---

## Architecture: WebSocket Three-Layer Separation

1. **`services/websocket.service.ts`** — raw WebSocket class with no React dependency: handles connection lifecycle, message serialisation, send/receive, and reconnection logic
2. **`hooks/useWebSocket.ts`** — React hook that wraps the service: ties the connection to the component lifecycle, exposes connection status and a typed send function
3. **`hooks/useMultiplayer.ts`** — game-level hook built on top of `useWebSocket.ts`: translates incoming WebSocket messages into room and opponent game state, and exposes methods for sending lock-event payloads

## Structural Principles

These are strict rules. Every file in your output must comply with all of them. If you deviate from any rule, you must explicitly flag the deviation in the Change Log and justify it.

- Every reusable component lives in its own named folder containing `ComponentName.tsx`. A `.module.css` file is included alongside the `.tsx` file if and only if that component applies component-scoped styles — omit it for purely structural or behavioural components that have no styles of their own. A component is **private** if it is imported only by a single parent component — private sub-components may be co-located inside their parent's folder as `SubName.tsx` and, if it has styles, `SubName.module.css`. Private sub-components must not be imported from anywhere else. If a private sub-component gains a second importer, it must be promoted to its own named folder.
- Only the four folder groups explicitly listed here export members via a barrel `index.ts`: `components/ui/`, `components/game/`, `components/layout/`, and `pages/`. A barrel file is one that re-exports members from multiple sibling files — a single-file module entry point such as `router/index.tsx` is not a barrel and is not subject to this rule. No other folder (`context/`, `hooks/`, `utils/`, `services/`, `types/`, `constants/`) gets a barrel file.
- Hooks contain only React logic (state, effects, refs) — all pure game calculations belong in `utils/`.
- All types that are exported from any file must be defined in `types/`. Types that are purely internal to a single file and never exported may be defined locally in that file.
- No hardcoded strings or magic numbers in components or hooks — all values live in `constants/`.
- Route paths are defined once in `constants/routes.constants.ts` and imported everywhere they are needed.
- Default keybinds are defined once in `constants/keybinds.constants.ts` and imported everywhere they are needed — never hardcoded in components or hooks.
- All colours, spacing, font sizes, and timing values are CSS variables declared in `styles/global.css` — no raw values in any `.module.css` file.
- Naming conventions: `PascalCase` for components and types, `camelCase` for hooks, utils, services, and constants files. Tooling-generated and tooling-configured files (`vite-env.d.ts`, `vite.config.ts`, `eslint.config.js`, etc.) are exempt from this convention.
- `localStorage` is only ever read or written inside `context/SettingsContext.tsx` — no other file touches it.
